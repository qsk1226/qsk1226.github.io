hashMap的容量为什么是2的幂次方

2020/10/04 posted in  数据结构与算法
Tags: 

源码中他们采用了延迟初始化操作，也就是table只有在用到的时候才初始化，如果你不对他进行put等操作的话，table的长度永远为"零"。

在HashMap中主要有两个函数保证了它的长度为 2 的 n 次方

  • tableSizeFor()
  • resize()

HashMap的 hash 算法是怎么实现的

jdk8中，通过hashCode的高16位与低16位进行异或操作

(key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16)

主要是高位也参与hash计算的话，可以提高hash算法的质量(只要 hashCode 有一位不一样，整个hash的结果就会不一样)和效率，降低 hash 冲突的发生。

为什么要保证 capacity 是2的次幂呢

在HashMap的存取过程中我们会看到这样的操作

static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}

HashMap是根据key的hash值决策key放入到哪个桶（bucket）中，通过 tab=[(n - 1) & hash] 公式计算得出。其中tab是一个哈希表。

当容量一定是2n时，h & (length - 1) == h % length, 我们知道 在计算机的世界中，所有的一切都是二进制的，所以通过位运算去模要比10进制的取模要效率高很多。

在get方法实现中，实际上是匹配链表中的 Node[] tab 中的数据。
(n - 1) & hash 实际上是计算出 key 在 tab 中索引位置，当key的hash没有冲突时，key在HashMap存储的位置就是匹配的node中的第一个节点。如果hash有冲突，就会在node里面节点中查询，直至匹配到相等的key。

因为 n 永远是2的次幂，所以 n-1 通过 二进制表示，永远都是尾端以连续1的形式表示（00001111，00000011），当(n - 1) 和 hash 做与运算时，会保留hash中 后 x 位的 1，例如 00001111 & 10000011 = 00000011

这样做有几个好处：

  • &运算速度快，至少比%取模运算块
  • 能保证 索引值 肯定在 capacity 中，不会超出数组长度
  • (n - 1) & hash，当n为2次幂时，会满足一个公式：(n - 1) & hash = hash % n
